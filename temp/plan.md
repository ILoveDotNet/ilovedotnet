# ILoveDotNet MCP Server — Implementation Plan

> **Goal:** Build an MCP (Model Context Protocol) server that serves knowledge from ilovedotnet.org so that AI coding assistants (Claude Code, GitHub Copilot) generate code that follows Abdul Rahman's exact patterns, style, and conventions as documented in the blog.

---

## Repo Context

| Item | Detail |
|---|---|
| Repo | `https://github.com/ILoveDotNet/ilovedotnet` |
| Site | `https://ilovedotnet.org` (Blazor WASM → GitHub Pages) |
| Blog content | `*DemoComponents/Pages/Blogs/*.razor` or `*DemoComponents/<slug>/*.razor` files |
| Table of Contents | `SharedModels/` — C# classes with structured blog metadata |

## Phase 1 — Content Extraction Pipeline

### Goal
Parse all `*DemoComponents/Pages/Blogs/*.razor` files and extract structured knowledge chunks into `MCPServer/Data/knowledge-base.json`. This runs at CI time so the knowledge base is always in sync with new blog posts.

### New Project: `ContentExtractor/ContentExtractor.csproj`

```xml
<Project Sdk="Microsoft.NET.Sdk">
  <PropertyGroup>
    <OutputType>Exe</OutputType>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="AngleSharp" Version="*" />
  </ItemGroup>
</Project>
```

### Extraction Logic: `ContentExtractor/Program.cs`

**Inputs:** All `*DemoComponents/Pages/Blogs/*.razor` files in the solution root.

**For each `.razor` file, extract:**

| Field | Source in .razor |
|---|---|
| `slug` | File name (e.g., `REPRInASPNET.razor` → `repr-pattern`) |
| `title` | First `<h2>` or `<PageTitle>` tag |
| `channel` | Parent folder name (e.g., `WebAPIDemoComponents` → `web-api`) |
| `url` | Derived from blog slug: `https://ilovedotnet.org/blogs/{slug}` |
| `tags` | Meta tags or channel + topic keywords |
| `codeSnippets` | All `<pre><code>` blocks extracted as raw strings |
| `summary` | First `<p>` paragraph text |
| `keyPoints` | All `<li>` items inside summary/advantage sections |

**Output:** `MCPServer/Data/knowledge-base.json`

```json
[
  {
    "slug": "request-endpoint-response-repr-pattern-in-asp-net-webapi",
    "title": "Request Endpoint Response (REPR) pattern in ASP.NET WEB API",
    "channel": "web-api",
    "url": "https://ilovedotnet.org/blogs/request-endpoint-response-repr-pattern-in-asp-net-webapi",
    "tags": ["REPR", "web-api", "controller", "SRP", "architecture"],
    "summary": "REPR pattern organises each API endpoint as its own controller class with a single HandleAsync method...",
    "keyPoints": [
      "Each endpoint inherits from ControllerBase",
      "Single public method named HandleAsync",
      "Use SwaggerOperation Tags to group in Swagger UI",
      "Enforce with NetArchTest fitness tests"
    ],
    "codeSnippets": [
      {
        "label": "REPR Endpoint",
        "language": "csharp",
        "code": "namespace ILoveDotNet.Article;\n\n[ApiController]\npublic class Publish : ControllerBase\n{\n    [HttpPut(...)]\n    public ActionResult HandleAsync(PublishArticleCommand request) { ... }\n}"
      },
      {
        "label": "Fitness Test",
        "language": "csharp",
        "code": "public class OnlyOneHandleRule : ICustomRule { ... }"
      }
    ]
  }
]
```

### CI Integration

Add a step to `.github/workflows/build-blazor-app.yml` **before** the Blazor publish step:

```yaml
- name: Extract blog knowledge base
  run: dotnet run --project ContentExtractor/ContentExtractor.csproj
```

This ensures `knowledge-base.json` is always current before the MCP server is deployed.

## Phase 2 — MCP Server

### Project: `MCPServer/MCPServer.csproj`

```xml
<Project Sdk="Microsoft.NET.Sdk.Web">
  <PropertyGroup>
    <TargetFramework>net10.0</TargetFramework>
    <Nullable>enable</Nullable>
  </PropertyGroup>
  <ItemGroup>
    <PackageReference Include="ModelContextProtocol.AspNetCore" Version="*" />
    <PackageReference Include="Microsoft.Extensions.Hosting" Version="*" />
  </ItemGroup>
</Project>
```

> Use the official `ModelContextProtocol.AspNetCore` NuGet package (from Microsoft/Anthropic SDK).

### Entry Point: `MCPServer/Program.cs`

```csharp
var builder = WebApplication.CreateBuilder(args);

builder.Services.AddSingleton<KnowledgeBaseService>();
builder.Services
    .AddMcpServer()
    .WithHttpTransport()
    .WithTools<PatternGuideTool>()
    .WithTools<TestGuideTool>()
    .WithTools<ArchitectureRulesTool>()
    .WithTools<LearningPathTool>()
    .WithTools<SearchBlogsTool>();

var app = builder.Build();
app.MapMcp();
app.Run();
```

### Service: `MCPServer/Services/KnowledgeBaseService.cs`

- Loads `knowledge-base.json` on startup into memory
- Exposes methods:
  - `GetByTag(string tag)` — returns all entries matching a tag
  - `GetByChannel(string channel)` — returns all entries in a channel
  - `Search(string query)` — simple full-text search across title, tags, keyPoints, summary
  - `GetBySlug(string slug)` — returns a single exact entry

---

### Tools

#### Tool 1: `PatternGuideTool.cs`

**MCP Tool Name:** `get_pattern_guide`

**Description:** Returns Abdul Rahman's implementation guide and code templates for a given .NET pattern as documented on ilovedotnet.org.

**Input:**
```json
{ "pattern": "REPR" }
```

**Logic:**
1. Call `KnowledgeBaseService.GetByTag(pattern)`
2. Find the best matching entry
3. Return an instructional response that includes:
   - What the pattern is (from `summary`)
   - Key rules to follow (from `keyPoints`)
   - Exact code template (from `codeSnippets[0]`)
   - Blog URL for reference

**Output format (returned to AI):**
```
## Abdul Rahman's {pattern} Pattern Guide

### Rules (from ilovedotnet.org)
1. {keyPoint1}
2. {keyPoint2}
...

### Code Template
```csharp
{codeSnippet}
```

### Reference
{url}
```

---

#### Tool 2: `TestGuideTool.cs`

**MCP Tool Name:** `get_test_guide`

**Description:** Returns Abdul Rahman's unit and integration test style for a given topic as documented on ilovedotnet.org.

**Input:**
```json
{ "topic": "web-api" }
```

**Logic:**
1. Call `KnowledgeBaseService.GetByChannel(topic)` filtered to TDD/Testing channel entries
2. Return xUnit test style, including:
   - Test class structure
   - `ITestOutputHelper` usage
   - AAA (Arrange, Act, Assert) pattern
   - Naming conventions

---

#### Tool 3: `ArchitectureRulesTool.cs`

**MCP Tool Name:** `get_architecture_rules`

**Description:** Returns NetArchTest fitness test rules that Abdul Rahman uses to enforce architecture for a given topic.

**Input:**
```json
{ "topic": "web-api" }
```

**Logic:**
1. Search knowledge base for entries tagged with `fitness-test`, `NetArchTest`, or `architecture`
2. Return the full fitness test code snippets from the REPR blog (OnlyOneHandleRule, OnlyOneActionResultRule)

---

#### Tool 4: `LearningPathTool.cs`

**MCP Tool Name:** `get_learning_path`

**Description:** Returns the ordered list of articles Abdul Rahman recommends for learning a .NET topic, sourced from the ilovedotnet.org Table of Contents.

**Input:**
```json
{ "channel": "web-api" }
```

**Logic:**
1. Read from `SharedModels` TableOfContents data (loaded at startup)
2. Return ordered article list with titles and URLs
3. Highlight prerequisite articles (e.g., SOLID before REPR)

---

#### Tool 5: `SearchBlogsTool.cs`

**MCP Tool Name:** `search_blogs`

**Description:** Full-text search across all ilovedotnet.org blog content. Use this to find relevant articles before generating code.

**Input:**
```json
{ "query": "middleware pipeline" }
```

**Logic:**
1. Call `KnowledgeBaseService.Search(query)`
2. Return top 5 matching entries with title, summary, url, and tags
3. AI can then call `get_pattern_guide` on relevant results

## Phase 3 — Extend AGENTS.md

Replace/extend the existing `AGENTS.md` at the repo root with the following standing instructions. This file is automatically read by Claude Code, GitHub Copilot agents, and Codex.

```markdown
# ILoveDotNet AI Coding Agent

This repository uses the ILoveDotNet MCP server as the authoritative source
for all code style, patterns, and conventions. Before generating any .NET code,
always consult the MCP tools below.

## MCP Server
URL: https://mcp.ilovedotnet.org/sse

## Mandatory Rules

### Web API
- ALWAYS use REPR pattern. Call `get_pattern_guide("REPR")` before writing any controller.
- Each endpoint = one class inheriting ControllerBase.
- Single public method named `HandleAsync`.
- Group Swagger docs with `[SwaggerOperation(Tags = new[] { "..." })]`.
- NEVER add more than one action method to a controller class.

### Testing
- ALWAYS use xUnit.
- ALWAYS inject `ITestOutputHelper` via constructor.
- ALWAYS follow AAA (Arrange, Act, Assert) with comments.
- Call `get_test_guide("web-api")` before writing any test.

### Architecture Fitness Tests
- ALWAYS include NetArchTest fitness tests for architecture rules.
- Call `get_architecture_rules("web-api")` to get the exact rules.

### General
- ALWAYS call `search_blogs("{topic}")` first to find if Abdul Rahman has
  written about the topic before generating code from scratch.
- Follow SOLID principles. SRP is non-negotiable.
- Use constructor injection only. No service locator pattern.
- Prefer `sealed` classes for endpoint controllers.
```

## End-to-End Flow (After All Phases)

```
User: "write an Orders API controller"

Claude Code / Copilot:
  1. Reads AGENTS.md → knows to call MCP before generating
  2. Calls search_blogs("API controller")
     ← MCP returns REPR article, TDD article, NetArchTest article
  3. Calls get_pattern_guide("REPR")
     ← MCP returns: rules + Publish endpoint template + URL
  4. Calls get_test_guide("web-api")

  5. Calls get_architecture_rules("web-api")
     ← MCP returns: OnlyOneHandleRule + OnlyOneActionResultRule

Generates:
  ✅ /Orders/CreateOrder.cs       → REPR endpoint, sealed, HandleAsync
  ✅ /Orders/CreateOrderTests.cs  → xUnit, ITestOutputHelper, AAA
  ✅ /FitnessTests/ControllerTests.cs → NetArchTest rules enforced
  ✅ SwaggerOperation Tags = ["OrderEndpoints"]
  ✅ Exactly how Abdul Rahman would write it

## Key Design Decisions

1. **No vector database / embeddings.** Your `.razor` files are already structured. Simple tag + full-text search is sufficient and keeps the stack pure .NET with zero external dependencies.

2. **Instructional responses, not raw content.** Each MCP tool wraps content in explicit rules ("Abdul Rahman's REPR pattern requires...") so the AI doesn't just see code — it sees *how to apply* the code.

3. **CI-driven knowledge base.** `knowledge-base.json` is regenerated on every push. When you publish a new blog post, the MCP server automatically knows about it within minutes.

4. **AGENTS.md as the forcing function.** The MCP is only useful if the AI knows to call it. `AGENTS.md` makes MCP tool calls mandatory, not optional.

5. **One solution, one repo.** `MCPServer` and `ContentExtractor` live inside the existing `ILoveDotNet.slnx` solution. No new repositories needed.
